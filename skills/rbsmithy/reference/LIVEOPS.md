# LiveOps and update planning

Use this file when planning releases, events, limited-time content, daily rewards, seasonal loops, or post-launch improvements.

## Philosophy

LiveOps is not random content spam. It is a measured cycle:

`Design -> Build -> Test -> Measure -> Release -> Learn -> Iterate`

Each update needs a reason:

- increase day-1 retention;
- improve tutorial completion;
- reduce boredom in the mid-game;
- add social goals;
- stabilize economy;
- fix performance;
- add fair monetization.

## Update types

### Foundation update

Fixes core pain:

- onboarding;
- UI readability;
- save reliability;
- performance;
- combat feel;
- input responsiveness.

### Content update

Adds new playable material:

- island;
- questline;
- NPCs;
- ship cosmetic set;
- activity;
- boss encounter;
- resource chain.

### Event update

Time-limited or seasonal content:

- festival;
- global boss;
- double resource weekend;
- community goal;
- limited cosmetics;
- rotating shop.

### Balance update

Changes numbers and pacing:

- resource rewards;
- cooldowns;
- stamina values;
- shop prices;
- durability loss;
- enemy health.

## LiveOps calendar fields

Every calendar entry should include:

- update name;
- release date/window;
- player-facing promise;
- systems touched;
- telemetry to watch;
- risk level;
- rollback plan;
- test plan;
- release notes draft.

## Claude output requirements

When the user asks for update planning, produce:

1. update goal;
2. player-facing summary;
3. production scope;
4. required systems;
5. analytics events;
6. risk list;
7. QA checklist;
8. release notes.
